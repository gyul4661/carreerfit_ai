# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문

## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.

---

## 프로젝트 전체

### Q1. 이 서비스가 해결하는 문제는 무엇인가?

CareerFit AI는 대학생이나 취업 준비생이 자신의 전공, 보유 스킬, 관심 직무를 기준으로 어떤 역량을 더 준비해야 하는지 판단하기 어려운 문제를 해결하기 위한 서비스이다.

채용공고마다 요구하는 기술, 우대사항, 직무 설명이 다르기 때문에 사용자가 직접 여러 공고를 비교해야 한다.

CareerFit AI는 채용공고 데이터를 기반으로 사용자의 현재 역량과 목표 직무 사이의 차이를 분석하고, 포트폴리오 개선 방향을 제안한다.

---

### Q2. RAG를 사용하지 않으면 어떤 문제가 생기는가?

RAG를 사용하지 않으면 AI가 실제 채용공고 데이터를 참고하지 않고 일반적인 답변만 생성할 가능성이 높다.

예를 들어 사용자가 “반도체 공정 엔지니어가 되려면 무엇을 준비해야 하나요?”라고 물었을 때, 실제 공고에 나온 요구 역량이 아니라 모델이 알고 있는 일반적인 조언만 답할 수 있다.

RAG를 사용하면 ChromaDB에서 관련 채용공고를 먼저 검색하고, 그 검색 결과를 Gemini에게 함께 전달하기 때문에 공고 데이터에 근거한 답변을 만들 수 있다.

---

### Q3. 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?

가장 어려웠던 부분은 ChromaDB 검색 결과와 Gemini 답변 생성을 연결하는 과정이었다.

처음에는 단순히 사용자의 입력만 LLM에 전달했지만, 이후 사용자의 전공, 스킬, 관심 직무를 하나의 query로 만들고 ChromaDB에서 관련 공고를 검색하도록 수정했다.

검색된 공고 내용을 context로 Gemini에 전달하고, 응답에는 answer와 sources를 함께 반환하도록 구성하여 RAG 기반 분석 흐름을 완성했다.

---

## 데이터 관련

### Q1. 왜 이 데이터를 선택했는가?

CareerFit AI는 사용자의 전공, 보유 스킬, 관심 직무를 바탕으로 취업·공모전 방향을 추천하는 서비스이다.

따라서 실제 사용자가 진로와 포트폴리오 방향을 판단할 때 참고할 수 있는 채용공고 데이터를 선택했다.

채용공고에는 회사명, 직무명, 요구 역량, 우대사항, 설명 등이 포함되어 있어 RAG 검색과 AI 답변 생성의 근거 데이터로 사용하기 적합하다.

---

### Q2. 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?

현재 프로젝트에서 사용한 데이터는 실습용 CSV 데이터이기 때문에 실제 기업의 최신 채용공고와 완전히 일치하지 않을 수 있다.

또한 채용공고의 요구 스킬이나 우대사항은 기업의 채용 시점에 따라 바뀔 수 있으므로 시간이 지나면 오래된 정보가 될 수 있다.

따라서 실제 서비스로 확장하려면 최신 공고를 주기적으로 수집하고, 데이터 출처와 업데이트 날짜를 함께 관리해야 한다.

---

### Q3. SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?

SQLite는 채용공고 데이터를 표 형태로 구조화해서 저장하고 조회하기 위해 사용했다.

회사명, 직무명, 요구 스킬, 마감일 같은 정형 데이터를 관리하기에 적합하다.

ChromaDB는 사용자의 질문과 채용공고 내용을 의미 기반으로 검색하기 위해 사용했다.

예를 들어 사용자가 “반도체 공정 엔지니어에 필요한 역량을 알려줘”라고 질문하면, 단순 키워드가 완전히 일치하지 않아도 의미가 비슷한 공고를 찾아올 수 있다.

즉, SQLite는 구조화 데이터 관리용이고, ChromaDB는 RAG 검색용이다.

---

### Q4. 결측치와 중복을 어떻게 처리했는가?

전처리 과정에서 CSV 데이터를 읽은 뒤 결측치와 중복 여부를 확인했다.

결측치가 있는 경우에는 필요한 컬럼이 비어 있는지 확인하고, 분석에 필요한 값이 없으면 빈 문자열이나 기본값으로 처리할 수 있도록 했다.

중복 데이터는 같은 공고가 여러 번 저장되어 검색 결과에 반복적으로 나타나는 것을 막기 위해 제거했다.

이를 통해 RAG 검색 결과가 불필요하게 중복되지 않도록 했다.

---

## RAG 관련

### Q1. RAG 구조에서 ChromaDB는 어떤 역할을 하는가?

ChromaDB는 채용공고 데이터를 벡터 형태로 저장하고, 사용자의 질문과 의미적으로 가까운 공고를 검색하는 역할을 한다.

사용자가 전공, 스킬, 관심 직무를 입력하면 이를 query로 만들고, ChromaDB에서 관련성이 높은 공고를 찾아 Gemini에게 context로 전달한다.

즉, ChromaDB는 AI 답변의 근거가 되는 문서를 찾아주는 검색 엔진 역할을 한다.

---

### Q2. n_results 값을 높이면 어떤 트레이드오프가 생기는가?

n_results 값을 높이면 더 많은 검색 결과를 가져올 수 있어 다양한 공고를 참고할 수 있다.

하지만 관련성이 낮은 문서까지 포함될 가능성이 높아지고, Gemini에게 전달하는 context가 길어져 응답 속도가 느려질 수 있다.

또한 불필요한 정보가 많아지면 답변의 초점이 흐려질 수 있다.

따라서 너무 많은 결과를 가져오기보다, 현재 프로젝트에서는 관련성이 높은 상위 몇 개의 공고를 사용하는 것이 적절하다.

---

### Q3. 검색 결과 품질을 어떻게 검증했는가?

사용자의 질문 예시를 직접 입력하고, ChromaDB가 반환하는 공고가 질문 의도와 맞는지 확인했다.

예를 들어 Python과 SQL이 필요한 데이터 분석 공고를 질문했을 때, 관련 스킬이 포함된 공고가 검색되는지 확인했다.

또한 `/analyze` API 응답에서 answer뿐만 아니라 sources를 함께 반환하도록 하여, AI가 어떤 공고를 참고했는지 검증할 수 있게 했다.

---

### Q4. Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?

Hallucination은 AI가 실제로 없는 정보를 마치 사실처럼 만들어내는 현상이다.

예를 들어 공고에 없는 자격요건이나 기업 정보를 AI가 임의로 생성하는 것이 이에 해당한다.

RAG는 AI가 답변을 생성하기 전에 실제 데이터에서 관련 문서를 검색하고, 그 내용을 근거로 답변하게 만든다.

따라서 모델이 완전히 추측으로 답하는 것을 줄이고, 검색된 공고에 기반한 답변을 생성하도록 도와준다.

---

## API 관련

### Q1. /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?

`/analyze`는 사용자의 전공, 보유 스킬, 관심 직무처럼 여러 입력 데이터를 서버로 보내야 한다.

GET 방식은 주로 데이터를 조회할 때 사용하고, 요청 데이터가 URL에 노출될 수 있다.

반면 POST 방식은 JSON body에 데이터를 담아 보낼 수 있어 사용자의 입력 정보를 전달하기에 더 적합하다.

따라서 `/analyze`는 POST 방식으로 구현했다.

---

### Q2. 응답 JSON에 sources를 포함시킨 이유는 무엇인가?

sources는 AI가 어떤 채용공고 데이터를 참고해서 답변했는지 보여주기 위해 포함했다.

AI 답변만 보여주면 사용자가 답변의 근거를 확인하기 어렵다.

sources를 함께 반환하면 회사명, 직무명 등 참고한 공고 정보를 확인할 수 있어 답변의 신뢰도를 높일 수 있다.

또한 RAG가 실제로 동작하고 있는지도 확인할 수 있다.

---

### Q3. /health 엔드포인트가 왜 필요한가?

`/health` 엔드포인트는 서버가 정상적으로 실행 중인지 빠르게 확인하기 위해 필요하다.

브라우저나 배포 플랫폼에서 `/health`에 접속했을 때 정상 응답이 오면 FastAPI 서버가 살아 있다는 것을 알 수 있다.

Docker나 Render 같은 배포 환경에서도 health check 용도로 사용할 수 있다.

## 보안 관련

### Q1. .env 파일을 GitHub에 올리면 어떤 일이 생기는가?

`.env` 파일에는 Gemini API Key 같은 민감한 정보가 들어갈 수 있다.

이 파일을 GitHub에 올리면 다른 사람이 API Key를 볼 수 있고, 무단으로 사용될 위험이 있다.

그 결과 API 사용량이 낭비되거나 비용 문제가 발생할 수 있다.

따라서 `.env` 파일은 `.gitignore`에 추가해서 GitHub에 올라가지 않도록 관리해야 한다.

---

### Q2. .env와 .env.example의 차이는 무엇인가?

`.env`는 실제 API Key와 환경변수 값이 들어 있는 파일이다.

반면 `.env.example`은 어떤 환경변수가 필요한지 알려주는 예시 파일이다.

`.env.example`에는 실제 키를 넣지 않고 `your_api_key_here`처럼 placeholder만 작성한다.

즉, `.env`는 비공개 파일이고, `.env.example`은 팀원이나 배포자가 설정값을 참고할 수 있도록 GitHub에 올려도 되는 파일이다.

## Docker 관련

### Q1. Docker를 사용하는 이유는 무엇인가?

Docker를 사용하면 개발 환경과 실행 환경을 컨테이너로 통일할 수 있다.

내 컴퓨터에서는 잘 되지만 다른 환경에서는 안 되는 문제를 줄일 수 있다.

또한 FastAPI 서버, 필요한 Python 패키지, 실행 명령을 Dockerfile에 정의해두면 다른 사람도 같은 방식으로 프로젝트를 실행할 수 있다.

Render 같은 배포 환경에서도 Docker 이미지를 기반으로 백엔드 서비스를 실행할 수 있다.

---

### Q2. Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

`EXPOSE`는 컨테이너 내부에서 사용하는 포트를 문서화하는 역할을 한다.

이 프로젝트에서는 FastAPI 서버가 8000번 포트에서 실행되므로 `EXPOSE 8000`을 작성했다.

`CMD`는 컨테이너가 실행될 때 기본으로 실행할 명령어를 의미한다.

이 프로젝트에서는 컨테이너 실행 시 FastAPI 서버가 자동으로 실행되도록 `uvicorn main:app` 명령을 CMD에 작성했다.

---

### Q3. Docker Image와 Container의 차이는 무엇인가?

Docker Image는 프로그램을 실행하기 위한 설계도나 실행 준비 파일이다.

Container는 그 Image를 실제로 실행한 상태이다.

비유하면 Image는 붕어빵 틀이고, Container는 그 틀로 실제로 구운 붕어빵이다.

하나의 Image로 여러 개의 Container를 만들 수 있다.

---

### Q4. Docker 컨테이너 실행 후 /health를 확인하는 이유는 무엇인가?

Docker 컨테이너가 실행됐다고 해서 애플리케이션이 정상적으로 동작한다고 바로 판단할 수는 없다.

컨테이너 내부에서 FastAPI 서버가 제대로 실행되고 있는지 확인하기 위해 `/health` 엔드포인트에 접속한다.

`/health` 응답이 정상적으로 오면 Docker 컨테이너 안에서 FastAPI 서버가 정상 실행 중임을 확인할 수 있다.
